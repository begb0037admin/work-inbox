## Architecture — Current Working State

| Component | Description |
|---|---|
| `fetch_inbox.py` | Runs on admin Windows machine. Phase 1: Outlook COM pulls 50 newest emails + all today/tomorrow calendar items. Phase 2: Anthropic API (claude-haiku-4-5) triages inbox → writes structured briefing.json. Phase 3: pushes briefing.json to GitHub via Contents API. Phase 3.5: AI triage proposes Command Centre tasks/updates from urgent+needs emails. Phase 3.6: auto-applies the updates to command-centre tasks.json (dated action entries, deduped via `data/triage_ledger.json`, daily backup enforced). Phase 5: pushes new-task suggestions to `data/inbox_suggestions.json` (rendered by the command-centre dashboard as draggable cards). |
| `index.html` | Single-file GitHub Pages dashboard. Fetches briefing.json from GitHub on load. Tick state loads from `data/ticks.json` and syncs on change via the Cloudflare Worker `cc-tasks-writer.kevinlelitte.workers.dev` (server-side PAT, no client secrets) — works from any machine. Oxford navy sidebar, light blue main, Inter font. No framework, no build step. |
| `open_email.py` | Registered openmail:// Windows protocol handler. Clicking a card opens the exact email in classic Outlook via COM EntryID. |
| `data/briefing.json` | Generated by fetch_inbox.py on every run. Pushed to GitHub. Dashboard fetches this on load. |

## Key Constraints
- `fetch_inbox.py` must always be pulled from GitHub before running — never run stale local copy
- Run script: `git fetch origin && git checkout origin/main -- fetch_inbox.py && python fetch_inbox.py`
- Inbox capped at 50 newest emails (newest-first sort) — guarantees morning emails always captured
- Calendar uses direct iteration with 30-day lookback + 6-day forward window — do NOT use `Restrict()` with `IncludeRecurrences` (returns empty on UK locale)
- ANTHROPIC_API_KEY and GITHUB_PAT stored as Windows User env vars — never in any file
- Model locked to claude-haiku-4-5 — Sonnet timed out on this inbox size
- `index.html` edits: always use binary `atob()`/`btoa()` approach — NEVER `TextEncoder` on file content (re-encodes em-dash bytes, re-garbles headings)
