## Architecture -- Current Working State

| Component | Description |
|---|---|
| `fetch_inbox.py` | Runs on admin Windows machine via Task Scheduler ONLY. Phase 1: Outlook COM pulls 50 newest emails + calendar. Phase 2: Anthropic API (claude-haiku-4-5) triages inbox → briefing.json in memory. Phase 3: pushes briefing.json to GitHub via Contents API. No local disk writes. |
| `index.html` | Single-file GitHub Pages dashboard. Fetches briefing.json from GitHub on load. Oxford navy sidebar, light blue main, Inter font. No framework, no build step. |
| `open_email.py` | Registered openmail:// Windows protocol handler. Clicking a card opens the exact email in classic Outlook via COM EntryID. |
| `data/briefing.json` | Generated by fetch_inbox.py on every run. Pushed to GitHub. Dashboard fetches this on load. |

## Critical Issue -- Manual Execution Broken
- Running fetch_inbox.py from PowerShell fails: COM error -2146959355
- Outlook runs in Session 2 (domain user), PowerShell runs in Session 1 (local admin)
- COM cannot cross session boundaries
- Workaround: use Task Scheduler only -- `schtasks /run /tn "WorkInbox-Briefing"`
- DO NOT attempt `python fetch_inbox.py` directly until resolved

## Key Constraints
- Task Scheduler auto-pulls fetch_inbox.py from GitHub before every run
- Manual trigger: `schtasks /run /tn "WorkInbox-Briefing"` (not python directly)
- Inbox capped at 50 newest unread, 30 read -- guarantees morning emails captured
- ANTHROPIC_API_KEY and GITHUB_PAT stored as Windows User env vars -- never in any file
- Model locked to claude-haiku-4-5 -- Sonnet timed out on this inbox size
- Task Scheduler requires elevated PowerShell to Register/Unregister tasks
- No local disk writes -- all output goes directly to GitHub
